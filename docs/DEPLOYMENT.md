# Deployment

VyroPortify is hosted across two providers, kept in sync via GitHub:

| Layer | Provider | Service | Cost |
|---|---|---|---|
| Frontend (Next.js) | **Vercel** | Hobby plan | $0 |
| Backend (FastAPI + Celery) | **Render** Web Service | Free | $0 |
| Postgres | **Render** PostgreSQL | Free (90-day expiry) | $0 |
| Redis | **Render** Key Value | Free | $0 |

**Production URLs**

| Surface | URL |
|---|---|
| App | https://vyroportify.vercel.app |
| API | https://vyroportify-api.onrender.com |
| API health probe | https://vyroportify-api.onrender.com/api/v1/health/ready |

## How it was deployed (reproducible)

### 1. Backend — Render Blueprint (one click)

The repo contains a [`render.yaml`](../render.yaml) blueprint. To recreate the
backend on any Render account:

1. https://dashboard.render.com/blueprints → **New Blueprint Instance**.
2. Connect the GitHub repo → Render auto-detects `render.yaml`.
3. Render provisions: 1 Postgres DB + 1 Key Value (Redis) + 1 Web Service
   (FastAPI + Celery sharing one container via `backend/start.sh`).
4. Fill in the secrets that ship with `sync: false` (see the env-var table
   below). The DB and Redis URLs auto-resolve via `fromDatabase` / `fromService`.

### 2. Frontend — Vercel CLI

The frontend is **not** in `render.yaml`. It's deployed via Vercel:

```bash
cd frontend
vercel link --yes --project vyroportify        # one-time
# Set env vars (production scope)
for KEY in NEXT_PUBLIC_API_URL NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY \
         CLERK_SECRET_KEY NEXT_PUBLIC_CLERK_SIGN_IN_URL \
         NEXT_PUBLIC_CLERK_SIGN_UP_URL NEXT_PUBLIC_CLERK_AFTER_SIGN_IN_URL \
         NEXT_PUBLIC_CLERK_AFTER_SIGN_UP_URL \
         NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY; do
  printf '%s' "$VALUE" | vercel env add "$KEY" production --force
done
vercel deploy --prod --yes
```

After the first push every commit to `main` auto-deploys on both providers.

## Post-deploy dashboard configuration

Three things live in third-party dashboards, not in code:

1. **Clerk Domains** — add `https://vyroportify.vercel.app` at
   https://dashboard.clerk.com → Domains. Without this, signups fail with CORS.
2. **Clerk Webhook URL** — at https://dashboard.clerk.com → Webhooks → edit
   your endpoint to `https://vyroportify-api.onrender.com/api/v1/auth/clerk-webhook`.
3. **Stripe Webhook endpoint** — at https://dashboard.stripe.com → Webhooks →
   Add endpoint:
   - URL: `https://vyroportify-api.onrender.com/api/v1/billing/webhook`
   - Events: `checkout.session.completed`, `customer.subscription.updated`,
     `customer.subscription.deleted`, `invoice.payment_succeeded`
   - Copy the `whsec_…` signing secret → set as `STRIPE_WEBHOOK_SECRET` on Render.

A walkthrough with screenshots is in [`docs/AUTH_SETUP.md`](AUTH_SETUP.md).

## Known limits of the current free-tier setup

| Constraint | Impact | Fix |
|---|---|---|
| **Render free Web Service sleeps after 15 min idle** | First request after sleep takes ~30-50s | Upgrade to Starter ($7/mo) |
| **Render free Postgres expires after 90 days** | DB gets deleted; data lost if not migrated | Upgrade to Basic ($7/mo) OR migrate to Neon/Supabase free tiers |
| **Render free Web Service is single-process** | `backend/start.sh` runs Uvicorn + Celery in one container; if uvicorn crashes Render restarts but Celery is silently lost until restart | Upgrade — separate Worker service ($7/mo) |
| **No HTTPS custom domain on free tier** | Stuck with `*.vercel.app` and `*.onrender.com` | Buy a domain → point CNAME → upgrade if needed |

When you outgrow free tier, the cleanest paid step is `render.yaml` plan changes:
`free` → `starter` on the web + a new dedicated `worker` service. Total: ~$14/mo.
